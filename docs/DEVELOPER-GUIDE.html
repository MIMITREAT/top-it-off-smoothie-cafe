00 Overview

This is a standalone static website for Top it off Smoothie Cafe (Grand Junction, CO), built on the Vivere Master Framework v2.0 — vanilla HTML5, CSS3, and JavaScript with zero front-end build step. It is hosted on Cloudflare Pages with a small serverless backend (Cloudflare Pages Functions + KV) powering a live daily flavor board.

Core Principle

No frameworks, no bundlers, no transpilers on the front end. What you write is what ships. If you're reaching for React, Tailwind, or a build pipeline — stop. This stack is intentionally simple, fast, and fully owned by the client.

03 The Versioning Rule ★ Most important

Cloudflare aggressively caches CSS and JS. To force browsers to load updated files, every HTML reference to styles.css and script.js uses a query-string version:

<link rel="preload" href="styles.css?v=7" as="style">
<link rel="stylesheet" href="styles.css?v=7">
...
<script src="script.js?v=4"></script>
Mandatory

When you edit styles.css, bump the version in every HTML file (?v=7 → ?v=8). Same for script.js. If you skip this, returning visitors will see the cached old version and your change "won't work." Current: styles.css?v=7 script.js?v=4

Files that reference these: index.html, about.html, menu.html, staff.html, 404.html. Grep before you ship: search the repo for ?v= and confirm all instances match.

04 CSS Conventions

Brand tokens (defined in :root)

TokenValueUse
--green-deep#14502ePrimary forest green — wordmark, dark sections
--accent#e8197dHot pink — CTAs, cursor, highlights
--accent-dark#c0125fPink for text on white (WCAG AA safe)
--accent-alt / leaf#6fbf44Fresh leaf green accents
Rule

Never hardcode brand hex values inline. Always reference the CSS variable (var(--green-deep)). The one exception: SVG wave fill attributes in HTML, which must be literal hex and must match the destination section's background exactly.

Section rhythm system

  • .section — white base section
  • .section--alt — green-tinted #e4f1da (visible alternation)
  • .section--warm — pink-white gradient (Mimi's tie-in)
  • .section--green — dark forest gradient, white text
  • .section--fruit-bg — section with faint photo background + overlay
Wave dividers

SVG .wave dividers bridge two sections. The wave's fill must equal the background of the section it flows into. If you change a section's background color, update the adjacent wave fill or you'll get a visible color seam.

Other conventions

  • Fluid type: use clamp(min, vw, max) for font sizes, not fixed px.
  • Spacing: 4px base engine — stick to multiples (0.25rem steps).
  • Mobile-first: base styles target mobile; use @media (min-width: …) to scale up.
  • One stylesheet: all CSS lives in styles.css. No inline <style> blocks except tiny one-off page tweaks.

05 JavaScript Conventions

All behavior lives in script.js (vanilla ES6+). Current modules:

  • Custom cursor — lerp-follow pink dot, desktop/pointer:fine only.
  • Card tilt — 3D perspective on mousemove (±10°), reset on leave.
  • Magnetic buttons — .btn--primary / .btn--green attract cursor.
  • View Transitions — intercept same-origin nav, document.startViewTransition(), silent fallback.
  • Scroll reveal — IntersectionObserver toggles .scroll-fade-up.
  • Live board fetch — GET /api/flavors-get, render homepage board.
  • Mobile nav — hamburger toggle with aria-expanded.
Rules

Every interactive enhancement must degrade gracefully — the site must work fully without JS. Guard pointer effects behind matchMedia('(pointer:fine)'). No external JS libraries — if a feature needs a library, raise it with Joe first.

06 Backend & KV

The live flavor board is powered by two Cloudflare Pages Functions backed by a KV namespace.

FunctionMethodPurpose
functions/api/flavors-get.jsGETPublic — returns current board for the homepage.
functions/api/flavors-set.jsPOSTStaff write. PIN auth + brute-force lockout present, currently disabled.

KV namespace

# wrangler.toml
[[kv_namespaces]]
binding = "[KV_BINDING_NAME]"
id = "[KV_NAMESPACE_ID]"

// KV keys, in general: a current-board key, a board metadata/timestamp key, and a staff-PIN key (unused while PIN auth is off). Exact key names live in the function source.

Before any staff turnover

PIN authentication is currently disabled on flavors-set.js — anyone with the staff URL can edit the board. A comment marks the restoration block. Re-enable it (and the per-IP brute-force lockout) before the cafe has public-facing staff churn. ~5 minutes of work.

Functions access KV via the binding: context.env.[KV_BINDING_NAME].get(...) / .put(...). They run at the edge — no server, no cold-start config.

07 Assets & Image Optimization

Non-negotiable

Never ship a raw PNG/JPEG hero or background. The original backgroundsmoothie.png was 24 MB and tanked load time to 121 seconds. Converted to WebP it's 131 KB. Always convert and serve WebP with a fallback.

Conversion (using sharp)

// run from project root where sharp is installed
sharp('assets/photo.jpg')
  .resize({ width: 1920 })
  .webp({ quality: 78 })
  .toFile('assets/photo.webp');

Markup pattern — always use <picture>

<picture>
  <source srcset="assets/acai.webp" type="image/webp">
  <img src="assets/acai.jpeg" alt="descriptive alt"
       loading="lazy" width="800" height="534">
</picture>
  • Keep originals in assets/ — they're the fallback and the re-export source.
  • Always set width + height to prevent layout shift (CLS).
  • Always write meaningful alt text (empty alt="" only for purely decorative images).
  • loading="eager" for above-the-fold (hero, logo); loading="lazy" for everything below.

08 SEO Standards

Every page ships with the full technical SEO foundation. New pages must include all of:

  • Unique <title> and <meta name="description">
  • <link rel="canonical"> pointing to the production URL
  • Open Graph tags (og:title, og:description, og:url, og:image, og:type)
  • Twitter Card tags
  • JSON-LD structured data — BreadcrumbList at minimum; homepage uses CafeOrCoffeeShop, menu uses Menu
  • Add the new page's URL to sitemap.xml
Domain not finalized

All canonical URLs, OG URLs, sitemap entries, and JSON-LD url fields currently point to the temporary [project].pages.dev URL. When the real domain is chosen, do a repo-wide find/replace across all files. Do not hardcode the pages.dev URL in new work without flagging it.

Staging SEO score

CF Pages preview URLs auto-inject X-Robots: noindex, so Lighthouse SEO shows ~69 on staging. This resolves to 100 on the production domain — it is not a bug.

09 Accessibility (WCAG AA)

The site passed a WCAG AA audit. Maintain it:

  • Contrast: normal text ≥ 4.5:1. Pink text on white uses --accent-dark (#c0125f), never --accent (fails AA).
  • Touch targets: interactive elements ≥ 44×44px (the staff dot, nav links, buttons).
  • Semantics: proper heading order, <nav>/<main>/<footer> landmarks, skip link.
  • ARIA: label icon-only buttons, set aria-expanded on toggles, aria-hidden on decorative elements.
  • Alt text: every meaningful image; alt="" for decorative only.
Test before ship

Run Lighthouse (or axe DevTools) on any page you touch. Accessibility regressions are bugs, not nice-to-haves — they carry legal (ADA) exposure for the client.

10 Security

Headers (_headers)

Global security headers are set: X-Frame-Options: SAMEORIGIN, X-Content-Type-Options: nosniff, Referrer-Policy, Permissions-Policy (geolocation/mic/camera off), X-XSS-Protection. Cache policy: assets immutable 1yr, CSS/JS 1 day, HTML must-revalidate.

Crawler rules (robots.txt)

  • Blocks 13 AI scrapers: GPTBot, ClaudeBot, CCBot, Google-Extended, PerplexityBot, Bytespider, anthropic-ai, meta-externalagent, Amazonbot, Applebot-Extended, cohere-ai, ChatGPT-User, and more.
  • Disallows /staff.html and /api/ for all crawlers.
Never commit secrets

No API tokens, no PINs, no credentials in the repo. The Cloudflare API token lives in Windows Credential Manager. KV namespace ID and account ID are identifiers (not secrets) and are fine in config — but never put a Cloudflare API token in any file.

Recommended before launch

Enable Cloudflare Bot Fight Mode. Re-enable staff PIN auth with a longer PIN. Confirm /api/ and /staff.html remain disallowed in robots.

Rules

  • Commit messages: conventional style — feat:, fix:, style:, docs:, chore:. Describe the why.

12 Deployment

Deploys are manual via Wrangler. Run from the project root.

Staging (always first)

npx wrangler pages deploy . --project-name [CF_PROJECT_NAME] --branch=staging --commit-dirty=true

Production (after staging review + approval)

npx wrangler pages deploy . --project-name [CF_PROJECT_NAME] --branch=main --commit-dirty=true
Standard ship sequence

1. Edit files → 2. Bump ?v= on CSS/JS if changed → 3. Test locally → 4. Deploy to staging → 5. Verify on staging URL → 6. Commit to git → 7. Deploy to main → 8. Verify production.

The client report (proposal/) is a separate CF Pages project ([REPORT_PROJECT_NAME]) — deploy from inside that folder, not the root.

13 Do's & Don'ts

✓ Do

  • Bump ?v= when editing CSS/JS
  • Convert images to WebP before shipping
  • Use brand CSS variables, not hex literals
  • Deploy to staging before main
  • Keep all CSS in one file, all JS in one file
  • Write semantic, accessible HTML
  • Set width/height + alt on images
  • Match wave fills to destination backgrounds
  • Test with Lighthouse after changes

✗ Don't

  • Add React, Vue, Tailwind, or a bundler
  • Ship raw multi-MB images
  • Hardcode brand hex inline
  • Commit node_modules or lh-*.json
  • Put API tokens or PINs in the repo
  • Deploy straight to main untested
  • Forget the cache-bust version bump
  • Hardcode the pages.dev URL in new work without flagging
  • Break graceful no-JS degradation

15 Vivere Master Framework Canon

This project is one instance of the Vivere Master Framework v2.0 — the agency-wide standard for every static build. Sections 00–14 above are how those rules apply here; this section is the canonical framework itself, reproduced so any dev understands the system end to end. Where the live-board backend (KV + Functions) extends the base framework, it's noted.

Framework Identity

A production-ready static-site system: a complete CSS design system, accessibility-first HTML patterns, a Cloudflare Pages deploy pipeline, full SEO infrastructure, and GA4 conversion tracking — all pre-wired. Stack is HTML5 / CSS3 / Vanilla JS on Cloudflare Pages. Target: 90+ Lighthouse, ~60s deploys.

Hard Rules — Never Break

① No React, Vue, Webpack, WordPress, Squarespace, or any server-side framework. ② No magic numbers — spacing only on the 4px grid (4, 8, 16, 32, 64px). ③ No hard-coded hex in elements — always CSS variables. ④ No jQuery or any JS animation library. ⑤ No blocking scripts — every <script> uses defer. ⑥ Max 2 font families per site.

Core Technical Rules

  • Typography: always fluid — font-size: clamp(min, preferred, max).
  • Images: hero = loading="eager" fetchpriority="high"; all others = loading="lazy". Always explicit width, height, alt.
  • Scripts: all <script> use defer. GA4 loads after paint via window.addEventListener('load', …).
  • Fonts: preconnect + async-load pattern, font-display: swap, max 2 families.
  • Layout: glassmorphism = rgba(255,255,255,0.98) + backdrop-filter: blur(12px). Bento grid = 1 → 2 → 4 col at 640px / 1024px.
  • Animations: IntersectionObserver only; add .visible / .scroll-fade-up class. No JS animation libs.
  • Marking: editable sections <!-- ✏️ EDIT: description -->; TBD content <!-- ✏️ PLACEHOLDER: description -->.

Accessibility Canon

  • .skip-link on every page; scroll-margin-top: 88px on #main-content.
  • aria-label on every <nav>; aria-labelledby on every <section>.
  • Every <input>/<textarea> has an associated <label>.
  • One <h1> per page; logical hierarchy h1 → h2 → h3, no skips.
  • Hamburger button keeps aria-expanded synced; closes on outside-click, link-click, Escape.

Standard Head Skeleton (per page)

Every page's <head> carries: charset + viewport, unique meta description (~155 chars), theme-color, unique <title> (<60 chars), canonical, full Open Graph + Twitter card, favicon stack, preloaded styles.css, async Google Fonts, GA4 (after-paint loader), and JSON-LD structured data. Update title / description / canonical / OG uniquely on each page.

Build Workflow — Cursor Builds, Claude Verifies

Cursor — Build Phase

  • Run the 11 sequential prompts in Agent (Ctrl+I), in order
  • Writes all code, multi-file edits
  • Pushes to staging after each major chunk
  • Does NOT auto-commit/push — you control git

Claude Code — Verify Phase

  • Audits all files against framework rules
  • Catches missed ARIA, hard-coded hex, broken hierarchy, leftover placeholders
  • Handles git commit/push/merge — only on explicit approval
  • Persistent memory across sessions

The 11 prompts cover, in order: ① framework init → ② project config → ③ homepage build → ④ SEO + structured data → ⑤ additional pages → ⑥ GA4 integration → ⑦ EmailJS → ⑧ SEO audit → ⑨ performance audit → ⑩ accessibility audit → ⑪ final pre-launch check.

Staging → Preview → Live

# 1. Local — nothing live yet
preview locally, hard-refresh between changes

# 2. Push to staging → CF builds preview URL (~60s, shareable)
git add . ; git commit -m "describe change" ; git push origin staging

# 3. Review on the preview URL (desktop + mobile), get approval

# 4. Merge to main → goes live (~60–120s)
git checkout main ; git merge staging ; git push origin main
Branch strategy

main = production (auto-deploys to live domain). staging = preview (auto-deploys to a CF preview URL). Never merge untested code directly to main.

Cloudflare 5-Step Setup (per new site)

  1. Add domain to Cloudflare — copy CF nameservers → paste at registrar → wait for propagation.
  2. Create GitHub repo — lowercase, no spaces, private recommended.
  3. Create CF Pages project — Connect Git, Framework: None, build command blank, output dir /.
  4. Configure branches — production = main; preview = all non-production branches.
  5. Connect custom domain (after first deploy) — add domain + www, verify green padlock.

GA4 Per-Site

Create a GA4 property → get G-[MEASUREMENT-ID]. Load after paint (window load event), config with send_page_view: true, anonymize_ip: true. In script.js add event tracking: form submission (generate_lead), phone clicks (a[href^=tel]), email clicks (a[href^=mailto]), CTA clicks (.btn--primary). Verify in GA4 Realtime before launch.

Credentials — never in the repo

CF account ID, GA4 measurement ID, EmailJS keys (service/template/public), and any API tokens are project credentials. Account/measurement IDs may live in config; API tokens never do — Wrangler auth lives in the OS credential store. Fill the .cursorrules Section 6 and a credentials template at project start; keep both out of version control.
